2.6 INTERACTIVE WORKFLOW PLAYER (hero centerpiece — build this as a real component)
============================================================
This directly mirrors the tab-strip demo player on withsurface.com's hero (the
CAPTURE / QUALIFY / ROUTE / RECOVER / NURTURE tabs that auto-advance and swap in a
full-bleed colored video panel). Build the SentinelGrid equivalent with these exact
mechanics:

a) TABS: a horizontal row of 4 tabs — "DETECT" · "PREDICT" · "RESPOND" · "RECOVER" —
   each with a small square icon (own accent color per tab, e.g. teal/amber/indigo/
   green) and an uppercase tracked label. Tabs sit in a light-gray strip above the
   video panel, same row width as the panel below.

b) ACTIVE STATE: exactly one tab is active at a time. The active tab is visually
   distinct (slightly darker background on its cell) and has a thin colored
   progress-fill bar directly beneath its label that fills left-to-right over the
   duration the panel is showing (~4-5 seconds), matching Surface's underline
   progress indicator exactly.

c) AUTO-ADVANCE: when a tab's progress bar finishes filling, auto-advance to the
   next tab in sequence (Detect → Predict → Respond → Recover → loop back to
   Detect). This must run continuously and automatically on page load — it should
   already be "playing" before the user touches anything.

d) MANUAL OVERRIDE: clicking any tab immediately jumps to that tab, resets its
   progress bar to 0%, and resumes auto-advance from there. Hovering/clicking
   pauses the auto-advance timer only while the user's cursor is over the strip
   (optional nice-to-have, not required).

e) CONTENT PANEL: below the tab strip, one large full-width panel (rounded corners,
   soft shadow, matches the "device frame" styling from Addendum 2 Section 3.2).
   The panel's background color shifts to match the active tab's accent color
   (soft/muted tint, not neon — e.g. a pale teal wash for Detect, pale amber for
   Predict, pale indigo for Respond, pale green for Recover). Inside the panel,
   show a short looping screen-capture/animation or illustrated diagram specific
   to that pipeline stage:
     — Detect: a simplified log/telemetry stream with one line highlighting red as
       an anomaly is flagged.
     — Predict: a simple confidence-meter or belief-score bar climbing as the
       Bayesian model updates.
     — Respond: an agent icon executing a contained action inside a sandboxed box
       outline.
     — Recover: a checklist/timeline ticking off items with a final green checkmark.
   Cross-fade (300ms) between panels when the active tab changes — no hard cuts.

f) TRANSITION BEHAVIOR: switching tabs should cross-fade both the panel content
   and its background tint together, not just swap content abruptly — mirrors the
   smooth, calm motion language from Addendum 2 Section 5.

g) WHY THIS MATTERS: this component is the single most important "wow" moment on
   the landing page — it's the first thing a judge interacts with before they ever
   click "LAUNCH LIVE DEMO," and it should visually preview the same Detect/
   Predict/Respond/Recover pipeline that the full simulation demonstrates, so the
   hero effectively acts as a compressed, always-playing trailer for the real demo.

h) REPLIT NOTE: implement with simple local component state (`activeTab`,
   `progress`) and a `setInterval`/`requestAnimationFrame` loop for the progress
   bar — no external video files required if time-constrained; an animated SVG/CSS
   illustration per tab is a perfectly good substitute for real screen-capture
   footage and is actually faster to build reliably in Replit.

14. PIXEL-SCHEMATIC ANIMATED ILLUSTRATION SYSTEM (replaces plain text/icons)
============================================================
This is the actual illustration language Surface uses in its own product clips —
dark bolted hardware panels, blocky pixel numerals, saturated flat-color tags, and
circuit-style routing lines — rather than plain icons or prose. Use this system for
every diagram referenced in Sections 2.6 (hero player), 4 (problem cards), and 5
(solution steps).

14.1 GRID BACKDROP: every diagram sits on a faint dot/line grid (1px lines, ~24px
     spacing, ~5% opacity) over a dark charcoal fill (#1E1E1E–#242424) for dark-
     mode diagrams, or a barely-visible warm-gray grid over off-white for light-
     mode diagrams (matches the Section 4/5 grid backdrops in the reference
     screenshots).
14.2 BOLTED PANELS: core/central elements (the "SentinelGrid," "belief state," or
     "audit trail" boxes) render as a dark rounded-rect panel with a thin lighter
     border and a small circular "bolt/screw" icon (half-shaded circle, like a
     flathead screw) in each of the top-left and bottom-right corners — this is the
     signature detail that reads as "technical hardware panel" rather than a
     generic card.
14.3 FLAT COLOR TAGS: small solid-fill rectangular tags carry every label
     ("TELEMETRY," "ANOMALY FLAGGED," "94," etc.) — sharp corners (not rounded),
     black or near-black pixel-style text, one flat saturated color per tag drawn
     from: amber-orange (#F2994A), yellow (#F2D94E), light green (#B9E88A), lavender
     (#A9A4FF), sky blue (#7FC8F8). No gradients or shadows on the tags themselves —
     flat fill only.
14.4 PIXEL NUMERALS: any big stat/count ("500," "94," "96") renders in a blocky,
     slightly-pixelated numeral style (a monospace/pixel-adjacent font such as
     "Press Start 2P" for numerals only, or a custom stepped-edge SVG numeral if a
     pixel font hurts legibility at large sizes) in the section's accent color.
14.5 STATUS ICONS: small octagon badges — green octagon + white checkmark for
     success/contained states, muted red-orange octagon + white "X" for
     missed/failed states — reused consistently across every diagram.
14.6 CONNECTOR LINES: thin gray (#8A8A85 on dark, #B9B9B4 on light) right-angle
     "circuit trace" lines connect elements — never diagonal, always routed in
     clean horizontal/vertical segments with rounded corner joins, matching the
     routing-diagram screenshots.
14.7 SIGNATURE MOTION PATTERNS (reuse across diagrams rather than inventing new
     motion per instance):
     — DECAY BOUNCE: an arc bounces along a horizontal dotted line, each
       successive bounce shorter, more transparent, and eventually dashed —
       communicates urgency/value draining away over time. Use for Problem Card 1.
     — COUNT-UP + BRANCH: a big pixel numeral ticks upward, then two gray lines
       branch downward to flat-color result tags — communicates one input
       resolving into a good and a bad outcome. Use for Problem Card 2 and Solution
       Step 2.
     — GHOSTED SILOS: two bolted panels at reduced opacity (~50%) with a connector
       line between them that never resolves/connects — communicates
       disconnection. Use for Problem Card 3.
     — LINE-DRAW ROUTING: connector lines animate in stroke-by-stroke (like an SVG
       `stroke-dashoffset` draw-on) from input tag to core panel to output tag(s) —
       communicates a live pipeline. Use for Solution Steps 1 and 4.
     — SPINNER CORE: a small ring of 8 pixel-dots rotates continuously inside a
       bolted panel — communicates active AI/agent processing. Use for Solution
       Step 3 and optionally as a loading state elsewhere on the page.
14.8 LOOPING RULE: every diagram loops continuously and automatically — none of
     these are one-shot animations the user has to trigger, and none are static
     end-states. This is the core fix requested: nothing in Problem or Solution
     should sit still.
14.9 REPLIT BUILD NOTE: these are all achievable as plain SVG + CSS (stroke-
     dashoffset for line-draw, CSS keyframe rotation for the spinner, opacity/
     translateY keyframes for the bounce-decay, a simple count-up via
     requestAnimationFrame for numerals) — no video assets needed. Build 4-5 small
     reusable React components (BoltedPanel, FlatTag, PixelNumeral, StatusOctagon,
     ConnectorLine) and compose the specific diagrams per section from those, rather
     than hand-building each diagram as a one-off.
